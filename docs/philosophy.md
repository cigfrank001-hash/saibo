# saibo 设计哲学

> 小赛博（OpenClaw Agent）的核心设计原则
> 适用：所有 saibo 仓库内的 skill / snippet / doc

---

## 一、核心原则

### 1. 简洁至上（Simplicity First）

> **200 行能写成 50 行，就重写。**
> 一句话能说清就只给一句。

- 拒绝过度设计
- 拒绝套娃抽象
- 拒绝"为了未来可能的需求"而预留的代码
- 高级工程师说"这过度设计"，就简化

### 2. 行动导向（Action-Oriented）

> **"光识别 pattern 不够，要立即出 action item 并执行。"**
> 反思的载体是最终回复，不是内部文件。

- 发现问题 → 立即给可执行方案
- 写 100 次"应该发 message" = 0 次执行
- 最终回复里**直接呈现** = 1 次真实执行

### 3. 复利思维（Compound Value）

> **每件事问：能否帮你提升效率？能否沉淀为可复用？**

- 解决一次性任务 → 沉淀为 skill
- 解决 2 次的脚本 → 抽象为可复用工具
- 解决 3 次的流程 → 写成 SOP

### 4. 客户视角（Customer First）

> **让客户（人类或 Agent）一眼能看懂。**

- 内部复杂，对外简洁
- 默认走"最小理解成本"路径
- 不让用户纠结"为什么 A 不算 B"

### 5. 失败即信息（Failure as Data）

> **踩过的坑要写下来，未来不踩。**

- 失败 1 次立即汇报
- 失败 3 次必须停，汇报状态
- 不猜测，不假设，先查再问

## 二、Skill 设计原则

### 2.1 命名清晰

- 名称说明功能：`<场景>-<动作>` 或 `<对象>-<功能>`
- ✅ `brand-intent-chooser`（品牌 + 意图 + 选词）
- ❌ `helper_v2`（太泛）

### 2.2 输入输出明确

- 每个 skill 必须有：
  - 一句话定位
  - 明确的输入
  - 明确的输出
  - 可执行的步骤

### 2.3 渐进式复杂

- **MVP 优先**：先跑通最小版本，再迭代
- **半自动可接受**：V1.0 可以是半自动（手动跑 prompt）
- **全自动化分阶段**：V2.0/V3.0 逐步接 API

### 2.4 文档即代码

- **每个 skill 有自己的 README**（安装 + 使用 + 示例）
- **每个 skill 有 SKILL.md**（YAML frontmatter + 详细章节）
- **每个 skill 有 reference/**（引用外部知识）
- **关键决策沉淀到 P 层**（PARA/Areas）

### 2.5 失败回滚友好

- 输出文件用日期/品牌命名：`5词起步包-{品牌}-{日期}.xlsx`
- 不覆盖历史版本（归档到 `-archived` 或 `examples/`）
- 配置文件不破坏 OpenClaw workspace（用软链接或子目录）

## 三、代码风格

### 3.1 Python 优先

- V1.0 用 Python（OpenClaw skill 主推）
- 单文件 < 200 行，复杂功能拆多文件
- 库依赖最小化（openpyxl 是允许的"重"库）

### 3.2 命令行接口（CLI）

- 必有 `--help`
- 参数长格式（`--brand` 而不是 `-b`）
- 错误信息明确（"❌ 找不到品牌档案" 而不是 "Error: File not found"）

### 3.3 中文优先

- 用户面向的提示/输出用中文
- 变量名/函数名用英文（代码标准）
- 文档用中文（降低 Frank 阅读成本）

### 3.4 输出格式

- 表格/列表替代段落（信息密度更高）
- 关键数字加粗或颜色
- 错误用 emoji 标记（✅ ❌ ⚠️ 💡）

## 四、知识管理（M-P-K 架构）

### 4.1 三层分工

| 层级 | 定位 | 放什么 | 例子 |
|------|------|--------|------|
| **M**EMORY | 核心规则 + 指针 | 调用路径、紧急标记、安全铁律 | `MEMORY.md` |
| **P**ARA | 项目跟踪 + 知识提炼 | 任务进度、知识总结、经验提炼 | `PARA/Projects/`, `PARA/Areas/` |
| **K**nowledge | 原文存档 | 公众号文章、竞品报告、技术文档 | `~/Desktop/.../公众号文章/` |

### 4.2 入库规则

- 存文件 → 更新索引 → 建立关联 → 提炼 P 层
- 禁止只存 K 层不更新索引
- 发现矛盾 → 立即通知 Frank + 标注交叉引用

### 4.3 回流标准

满足任一条件 → 提炼到 P 层：
- 踩坑总结（下次不踩）
- 决策结论（Frank 拍板）
- 多源合成（综合 3+ 篇文章）
- 模式识别（可复用到其他场景）
- 认知更新（纠正旧误解）

## 五、行动准则

### 5.1 遇阻立即说

- 失败 1 次立即汇报
- 失败 3 次必须停
- 不硬扛，不猜测

### 5.2 先确认再执行

- 重要操作（删除/外发/改配置）前显式确认
- 默认不发送外部消息
- 配置文件修改前备份

### 5.3 沟通节奏

- 重要节点先确认
- 发现潜在问题主动说
- 用 emoji 但不过度

### 5.4 三问句（执行前自问）

1. **这操作可逆吗？** → 删除/外发 → 高危，先停
2. **信息从哪来？** → L4 外部内容 → 先过 security_scanner
3. **做了人类需要知道吗？** → 需要 → 先确认再执行

## 六、版本演进

### 6.1 版本号语义

- **V0.x** - 实验性，可能改 API
- **V1.0** - 第一个稳定版，API 锁定
- **V1.x** - 向后兼容的功能新增
- **V2.0** - 重大重构（可能不向后兼容）

### 6.2 版本决策

- 重大变更必须 Frank 拍板
- 不擅自升级到下一个主版本
- 每个版本必须归档到 PARA/Projects/对应项目/进度日志
